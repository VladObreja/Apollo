## What Went Well

### Architectural Discipline Held
- **DI Protocol pattern** (`MarketDataClient`, `SMTPClient`, `EnvDataClient`) applied consistently across all 3 Epic 3 services with zero drift from the established pattern.
- **Fail-operational contract** maintained perfectly across Phases 4 and 5 — neither phase ever crashes `tick()` under SMTP failure, market data outage, or DB errors.
- **Double-blind isolation** in Story 3.3 is a genuine achievement: `CalibrationService` provably never joins `corpus_record`. The constraint is enforced by code structure, not convention.

### Code Review Caught Real Bugs
Story 3.1 review surfaced a URL injection risk (ticker not URL-encoded in Yahoo Finance URL) and a `_day_unix_range` timezone ordering bug that would have produced wrong day boundaries for non-UTC inputs. Both would have been silent production errors.

### Test Discipline
Test count grew from ~119 (start of Epic 3) to 255 (end). The discipline of writing unit tests for pure functions (Brier score, ECE, Wilson CI, `_day_unix_range`) with hand-verified math was the right call — those functions are subtle enough to get wrong.

### MCP Interface Complete
Three operational tools now exist: `configure_target`, `trigger_closure_ceremony`, `get_calibration_stats`. The admin interaction model (Vlad → Claude Code → MCP → Apollo) is fully implemented as designed.

## Challenges and Growth Areas

### Story 3.2 Complexity Spike
Story 3.2 (Closure Ceremony) had the heaviest review burden of the entire project: **16 patches + 2 decisions**. Story 3.1 had 7 patches; Story 3.3 had 4. The complexity was real — concurrent-tick serialization (advisory lock), SMTP failure atomicity, interval bypass logic, and the fail-operational double-layer were all non-trivial to get right simultaneously. The high patch count reflects genuine complexity, not carelessness.

### The `_unique_hash()` Pattern Was Re-Invented Twice
Both Story 3.2 and 3.3 debug logs record a `UniqueViolation` on `ix_corpus_record_raw_hash` when test helpers used static `"a"*64` hashes. The fix (`uuid4().hex * 2`) was independently re-discovered. This belongs in `tests/utils.py` as a shared utility.

### The 4 E2E Tests Never Got Fixed
`tests/e2e/test_epic2_e2e.py` — 4 tests written for Story 2.1 behavior — have been failing since Story 2.3 changed how quarantine handles `raw_email_bytes`. They were flagged in Story 3.1 completion notes and again in Story 3.2 notes. They were never fixed. Known-failing tests become noise that trains developers to ignore red.

### Recurring `IntegrityError` Catch — Three Code Reviews, Three Deferrals
The overly-broad `IntegrityError` catch in worker Phase 3 was flagged in Story 2.3, Story 3.1, and Story 3.2 code reviews. It was deferred each time. After three independent flags, this is not a matter of priority — it's a matter of scheduling it as an explicit story.

### Project Complexity Is Becoming Hard to Hold in One Head
Vlad's direct observation: by the end of Epic 3, the system has 5 phases in `tick()`, 4+ services, compartment guards, an append-only ledger, advisory locks, and a calibration engine. The code is well-organized, but there is no operator's manual, no architecture walkthrough document, and no SOP for daily operation. This gap matters for both continued development and real-world operation.

## Key Insights

1. **The DI Protocol pattern is this project's most reusable architectural win.** Every new service in Epic 3 followed it without discussion. It enabled clean testing, fail-operational isolation, and compartment enforcement simultaneously.

2. **Code review load is a signal, not just a checklist.** Story 3.2's 16-patch review was a warning that the story contained more interacting systems than any previous story. Future epics should consider splitting stories of similar density.

3. **Deferred items with 3+ independent flags are no longer truly deferred — they are unscheduled stories.** The `IntegrityError` catch is the primary example.

4. **The E2E test suite is currently unreliable as a signal.** 4 known-failing tests mean "all red" and "4 tests red" look the same. This should be fixed before real-world operation to preserve trust in CI.
