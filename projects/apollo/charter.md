# Apollo Charter

## Mission
Apollo is a local-first governing knowledge-and-operations system that ingests diverse sources, structures cross-domain understanding, and supports project execution through a single administrator-facing assistant coordinating specialized subsystems.

In shorter form:

**Apollo acquires knowledge, organizes meaning, and supports action.**

## Core purpose
Apollo is intended to function as both:
- a second brain
- an operational assistant

It must be able to:
- ingest heterogeneous information from many formats
- preserve provenance and project boundaries
- support cross-source analysis and synthesis
- help maintain ongoing regimens, protocols, and project workflows
- later host specialized modules such as Aurelius, Remote Viewing, and subliminal/pathworking generation

## Current strategic stance
Apollo is not to be treated as a random stack of tools.
It is to be treated as a coherent system with clear responsibility boundaries.

The human experience should be:
- one governing assistant
- one trusted command surface
- modular specialist capabilities underneath

## Design principles
1. **Local-first, cloud-augmented only when justified**
2. **One governing assistant, not one monolith**
3. **OTS-first, custom where Apollo needs unique control**
4. **Knowledge, interpretation, and action must remain distinct**
5. **Project boundaries must stay explicit**
6. **Provenance is mandatory**
7. **Backbone before theatrics**
8. **Human-readable files remain the primary control surface**

## Current scope
### In scope now
- project-aware ingestion and corpus organization
- normalized project control docs
- retrieval over Apollo and Aurelius planning material
- source-grounded querying through the running stack
- preparing a clean foundation for later operations

### In scope later
- regimen scheduling and outcome tracking
- RV tasking and session workflows
- audio/video generation workflows
- more advanced agent orchestration
- remote operator flows

### Explicitly out of scope for Phase I Alpha
- complex autonomous multi-agent councils
- advanced medical-style recommendation automation
- heavy UX polish before the backbone works
- universal ingestion of every format before the text-first loop is proven

## Success condition for Phase I Alpha
Apollo can ingest or use the existing clean planning corpus, index it in retrieval, and answer grounded questions about Apollo and Aurelius through a single operator-facing interface.

## Relationship to Aurelius
Aurelius is not separate from Apollo, but it is not identical to Apollo either.
Aurelius should be treated as the first serious operational module or program living on top of Apollo’s shared infrastructure.

## Relationship to Vault/Clean
`Vault/Clean/` is a valuable supporting corpus and evidence layer.
`projects/apollo/` is the canonical control layer for purpose, decisions, progress, and handoff.
