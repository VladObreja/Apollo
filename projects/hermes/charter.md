# Hermes Charter

## Mission
Hermes is a focused Apollo subproject for remote viewing and broader psi-functioning operations.

Its purpose is twofold:
- support psi-functioning research with disciplined operational tooling
- explore whether structured remote-viewing workflows can produce usable signal in financial and other target domains

In shorter form:

**Hermes manages tasking, capture, feedback, and analysis for psi operations.**

## Core purpose
Hermes is intended to function as both:
- a research support system for controlled psi-functioning work
- an operational platform for structured targeting, session handling, and outcome analysis

It must eventually be able to:
- manage task creation and controlled task release
- support blind or partially blind session workflows
- collect and store viewer/session data in structured form
- release feedback in a controlled way
- analyze sessions, viewers, target types, and outcomes over time
- later support market-oriented remote-viewing workflows, beginning with analysis and paper-trading style evaluation

## Relationship to Apollo
Hermes is not separate from Apollo’s infrastructure, but it is a distinct subproject with its own mission, operating rules, and project record.

Apollo remains the governing platform.
Hermes is a specialized module/program that lives on top of Apollo’s shared infrastructure and focuses specifically on psi-functioning workflows.

## Current strategic stance
Hermes must be built in small, useful chunks.

The guiding rule is:
- do not be ambitious but rubbish

That means Hermes should begin as a narrow, reliable backbone rather than a sprawling theory-machine.
The first goal is not to prove every metaphysical claim or automate every imagined workflow.
The first goal is to create a disciplined system that can run real sessions, preserve records, and support later analysis.

## Human actors
Hermes currently assumes the following human roles:
- **Project administrator:** the user, who defines scope, manages operations, reviews results, and decides escalation
- **Psionic assets / viewers:** one or more human operators who receive tasking, submit session data, and later receive feedback

Additional human roles may be added later, but Hermes should not depend on a large team in order to become useful.

## System components
Hermes is expected to use:
- locally run agents where practical, including use of the RTX 5070 Ti
- cloud/API models when justified
- this OpenClaw instance as the central coordination surface
- a remote interface so the administrator and viewers can interact with the system off-machine
- a database for session storage, metadata, and analysis
- later, controlled access to financial and other external data sources for target generation, feedback, and performance measurement

## Design principles
1. **Small chunks first**
2. **Research backbone before market automation**
3. **Structured records over vague impressions**
4. **Blindness and protocol discipline where required**
5. **Apollo remains the governing platform**
6. **Human review before consequential action**
7. **Paper and analysis before real-money execution**
8. **Human-readable control docs remain the primary continuity layer**

## Current scope
### In scope now
- Hermes project control and continuity docs
- definition of user roles and operating boundaries
- blind task creation and assignment workflows
- remote session submission/intake
- controlled feedback release
- structured session and target storage
- basic analysis of sessions, viewers, target categories, and results

### In scope later
- richer viewer analytics and historical comparison
- multiple viewer coordination
- protocol variants for different task types
- market-data ingestion for target generation and feedback
- paper-trading evaluation workflows
- more advanced local/cloud model routing

### Explicitly out of scope for Phase 0 Foundation
- live automated trading
- autonomous real-money execution
- overbuilt multi-agent hierarchies
- broad metaphysical theorizing without operational support value
- expansion into every possible psi modality before the first workflow works

## Success condition for Phase 0 Foundation
Hermes succeeds in its first phase when it can support a full end-to-end research loop for at least one administrator and one viewer:
- create a task
- deliver it remotely in a controlled way
- receive a session submission remotely
- store the data in structured form
- release feedback at the correct stage
- preserve the record for later analysis
- produce at least basic review outputs about what happened

## Operating bias
Hermes should prefer:
- repeatable workflows over mystical theater
- analyzable data over anecdotal drift
- explicit protocols over improvised memory
- gradual expansion over premature system sprawl

## Long-term intent
If the research backbone proves useful, Hermes may later expand into a serious operational environment for market-facing remote-viewing experiments and related analytical workflows.

That expansion must be earned by evidence.
It is not the starting assumption.
